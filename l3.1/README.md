# DelayedNotifier

Сервис отложенных и запланированных уведомлений, построенный на микросервисной архитектуре с использованием брокера сообщений. Позволяет регистрировать задачи на отправку уведомлений через различные каналы связи (Email, Telegram) как мгновенно, так и с заданной задержкой по времени.

---

## Технологический стек

* **Язык разработки:** Go (Golang)
* **Веб-фреймворк:** Gin Gonic (для HTTP API)
* **Брокер сообщений:** RabbitMQ (с гибкой маршрутизацией через Exchanges и Queues)
* **База данных:** PostgreSQL (для хранения истории и состояний уведомлений)
* **Кэширование и распределенные замки:** Redis
* **Логирование:** Zap Logger (структурированные логи)

---

## Архитектура транспортного слоя (RabbitMQ)

Для надежной доставки и изоляции логики обработки в брокере настроена следующая схема:
1. **Exchange (`delayed.notifier`):** Тип `topic`. Принимает сообщения от издателя (Publisher) из Go-приложения.
2. **Queue (`notification-queue`):** Очередь, в которую распределяются сообщения. Она связывается с обменником через **Binding** по ключу маршрутизации (`Routing Key`) `notification-queue`.
3. **Consumer (Воркер):** Фоновый Go-процесс, который слушает очередь, обрабатывает логику отправки во внешние API и, в случае сетевых сбоев, запускает политику повторных попыток (Exponential Backoff).

##️ Настройка конфигурации (`config.yaml`)

Перед запуском приложения убедитесь, что ваш файл конфигурации заполнен актуальными данными. Особое внимание уделите настройкам почтовых шлюзов.

```yaml
HTTP:
  ADDRESS: "localhost:8080"

POSTGRES:
  POSTGRES_USER: user
  POSTGRES_PASSWORD: user
  POSTGRES_HOST: localhost
  POSTGRES_PORT: 5432
  POSTGRES_DB: DelayedNotifier
  POSTGRES_DSN: postgres://user:user@localhost:5432/DelayedNotifier
  MAX_CONNS: 10
  MIN_CONNS: 5

REDIS:
  ADDRESS: 127.0.0.1:6379
  PASSWORD: ""
  MAX_MEMORY: 256mb
  POLICY: allkeys-lru

RABBITMQ:
  URL: amqp://guest:guest@localhost:5672/
  CONNECTION_NAME: DelayedNotifier
  EXCHANGE: delayed.notifier

TELEGRAM:
  BOT_TOKEN: "your_telegram_bot_token"

EMAIL:
  SMTP_PORT: 587             # 587 для STARTTLS (Gmail) или 465 для чистого SSL/TLS
  SMTP_HOST: smtp.gmail.com
  SENDER_EMAIL: "your_account@gmail.com"
  PASSWORD: "your_app_password_here" # Специальный 16-значный пароль приложения (не от аккаунта!)

```

> **Важно для Email:** Современные почтовые сервисы (Yandex, Gmail) требуют генерации отдельного **Пароля приложения** в настройках безопасности вашего аккаунта для авторизации по протоколу SMTP. Использование обычного пароля от почты приведет к ошибке авторизации.

## Подготовка окружения перед запуском

Выполните следующие шаги последовательно, чтобы подготовить проект к работе:

### 1. Скачивание Go-зависимостей

Убедитесь, что все внешние библиотеки (включая фреймворк `wbf`, `gin`, драйверы БД) загружены в ваш локальный кэш:

```bash
go mod tidy
go mod download
```

### 2. Запуск инфраструктуры (PostgreSQL, Redis, RabbitMQ)

Если у вас установлены Docker и Docker Compose, вы можете поднять всю необходимую инфраструктуру одной командой.

Создайте файл `docker-compose.yml` в корне проекта (если его ещё нет) и выполните:

```bash
docker-compose up -d
```

*Эта команда скачает и запустит контейнеры в фоновом режиме. Убедитесь, что порты `5432` (Postgres), `6379` (Redis) и `5672`/`15672` (RabbitMQ) свободны.*

### 3. Применение миграций базы данных

Для создания таблиц в PostgreSQL (таблицы `notifications` и др.) необходимо применить миграции.

Если вы используете утилиту `goose`, выполните:

```bash
goose -dir db postgres "postgres://user:user@localhost:5432/DelayedNotifier?sslmode=disable" up
```

*(Если у вас в проекте инициализация таблиц зашита прямо в код при старте `main.go`, то этот шаг пропустите — таблицы создадутся автоматически при первом запуске).*

### 4. Проверка доступности портов

Перед запуском Go-приложения вы можете проверить, что все сервисы успешно поднялись и слушают порты:

```bash
# Проверить Postgres
nc -zv localhost 5432

# Проверить Redis
nc -zv localhost 6379

# Проверить RabbitMQ
nc -zv localhost 5672

```

## Запуск проекта

### 1. Настройка Binding в панели управления RabbitMQ

Так как обменник использует тип `topic`, сообщения не попадут в воркер без явной привязки очереди.

1. Перейдите в веб-интерфейс управления брокером: `http://localhost:15672`.
2. Вкладка **Exchanges** -> выберите ваш обменник `delayed.notifier`.
3. В блоке **Add binding from this exchange** заполните поля:
* **To queue:** `notification-queue`
* **Routing key:** `notification-queue`


4. Нажмите кнопку **Bind**.

### 2. Очистка зависших очередей (при необходимости)

Если в брокере скопились старые сообщения с некорректными конфигурациями, очистите их, чтобы избежать цикличных ошибок воркера (`EOF` или `connection reset`):

1. В панели брокера перейдите на вкладку **Queues**.
2. Нажмите на очередь `notification-queue`.
3. Прокрутите вниз до секции **Purge** и нажмите **Purge Messages**.

### 3. Запуск Go-приложения

Из корневой директории проекта выполните команду запуска HTTP-сервера и фоновых процессов:

```bash
go run ./cmd/main.go
```

## Спецификация API (Эндпоинты)

### 1. Создать уведомление

* **URL:** `/notify`
* **Метод:** `POST`
* **Тело запроса (JSON):**

```json
{
  "recipient": "quuteo86@gmail.com",
  "channel": "email",
  "message": "Привет! Это тестовое уведомление из Go-воркера.",
  "scheduled_at": "2026-07-10T12:35:00Z"
}
```

### 2. Получить список всех уведомлений

* **URL:** `/notify`
* **Метод:** `GET`

### 3. Получить конкретное уведомление по ID

* **URL:** `/notify/:id`
* **Метод:** `GET`

### 4. Удалить/отменить уведомление

* **URL:** `/notify/:id`
* **Метод:** `DELETE`
